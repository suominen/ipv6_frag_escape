# IPV6_FRAG_ESCAPE Tracking — Claude Code Context

This repository contains a living tracking document for
**IPV6_FRAG_ESCAPE**, an unprivileged container-to-host-root escape in the
Linux kernel.  An arithmetic error in the IPv6 fragmentation path
(`__ip6_append_data()` in `net/ipv6/ip6_output.c`, with an IPv4 sibling
`__ip_append_data()` in `net/ipv4/ip_output.c`) undersizes an allocation
so the packet's own head object overflows into the trailing
`skb_shared_info`, giving control of `nr_frags` — a page use-after-free the
PoC turns into a Dirty-Pagetable primitive and a root shell via a
`core_pattern` overwrite.

The canonical fix is `736b380e28d0` (*ipv6: account for fraggap on the
paged allocation path*) plus its IPv4 sibling `eca856950f7c`, merged as
`38becddc` and first released in **v7.2-rc1**.  The bad accounting was
introduced dormant in v6.0 (`773ba4fe9104` / `8eb77cc73977`) and made
reachable in v6.6 by `ce650a166335`.  The **reachable window is kernels
6.6 through 7.1**.

There is **no CVE**.  Discovered by Massimiliano Oldani (`sgkdev`);
disclosed 2026-06-29 ([LinkedIn][writeup]) / 2026-06-30
([AlmaLinux][alma], [TuxCare][tuxcare], [Finnish NCSC][ncsc]).  PoC:
<https://github.com/sgkdev/ipv6_frag_escape>.

The rendered site is published at <https://kimmo.cloud/ipv6_frag_escape/>.

## What flips a verdict — the kernel backport, and only that

Unlike a userspace-gated chain, this bug lives entirely in the kernel and
the IPv6/IPv4 fragmentation path is always built in — there is **no
package-version reachability gate** (no cifs-utils-style second axis).  A
row's verdict is a single question:

- **Kernel** — is it inside the reachable window (6.6–7.1) *and* missing
  the `736b380e28d0` backport?  A kernel `≥ 7.2`, or `< 6.6`, or carrying
  the backport, is not exploitable.  This is what *Status* and *Fixed
  since* key on.

`CONFIG_INIT_ON_ALLOC_DEFAULT_ON` is **not** a second verdict axis — it
decides only *how bad* an unpatched in-window kernel is:

- **off** (EL10 default) ⇒ the PoC reaches **root** ⇒ `:x:`.
- **on** (Debian, Ubuntu, NixOS) ⇒ the freed slot is zeroed, so the same
  trigger only NULL-derefs ⇒ **denial of service only** ⇒ `:warning:`.

Record the `init_on_alloc` posture in the per-distro `###` prose, never as
a column.  The other PoC preconditions (LA57 active, world-readable BTF,
unprivileged userns, x86-64) are exploitation caveats for the *How the
chain works* / *Detection* prose, not verdict inputs.

The combined distribution-status table is the **single source** for each
row's kernel, *Fixed since*, and status.  Don't restate the table's columns
in prose or add a parallel per-release table.  Cell formatting: the
**Kernel** cell holds the version only (or `:grey_question:` when
unverified) — no "unpatched" / "patched" word, since *Status* and *Fixed
since* carry the verdict.  Label NixOS channels in the **Release** column
in friendly form (`Unstable`, `26.05`), not the raw `nixos-…` identifiers.
Keep the `{.distros}` block attribute on the line **immediately after** the
table (no blank line between) — it drives the `custom.css` rule that stops
the Release column from wrapping mid-label.

A per-distro `###` section is for **reader-facing** caveats that don't
fit the table (root-vs-DoS `init_on_alloc` posture, userns default,
EL10-family scope).  Keep tracking methodology out of it — which distros
are reference-only, how NixOS channels are structured, and similar are
agent guidance that belongs in this file, not the tracker.

## Routine run scope — frozen version snapshots

The per-distro **Kernel** versions in the table are **frozen seed
snapshots**, not live values: update a row's recorded version only when its
verdict actually changes.  A row cannot flip to fixed until the fix is
available to adopt — i.e. its kernel series carries :white_check_mark: in
the *Upstream fixed versions* table, **or** the distro cherry-picks
`736b380e28d0` independently.

As of the seed (2026-07-03) that has **not** happened for any stable
series: the fix reached mainline as v7.2-rc1 with **no `Cc: stable`**, so
6.6.y / 6.12.y / 7.0.y / 7.1.y are all still unpatched upstream, and the
only distro fix so far is AlmaLinux's independent cherry-pick
(`kernel-6.12.0-211.28.2.el10_2`, 2026-06-30).  So each run:

- Re-check the *Upstream fixed versions* table: has any in-window stable
  branch (6.6.y, 6.12.y, 7.0.y, 7.1.y) picked up the backport yet?  Search
  `~/src/linux/stable` for it (recipe below).  When a series flips, distro
  adoption becomes the live question for its rows.
- For a distro row, re-pull the distro's **kernel** version and compare:
  the kernel reaches the first-fixed release **or** a distro advisory /
  changelog ships the `736b380e28d0` backport ⇒ flip to :white_check_mark:,
  record the adopted kernel, set *Fixed since*.
- A version bump that stays **inside the vulnerable window without the
  backport** is **not** adoption ⇒ leave the row and its recorded version
  untouched.  Recording such a bump is version-only churn
  (`tracker: … kernel bump`) this tracker exists to avoid.
- Exception: a distro flipping its **default kernel series** (e.g.
  `proxmox-default-kernel` 2.1.0 moving PVE 9 from 6.14 to 7.0) *is*
  worth recording even when the verdict holds — update the table row,
  prose, and verification log **together**; a log-only update leaves the
  tracker self-inconsistent.
- Watch AlmaLinux (leading indicator) and Rocky/RHEL for the EL10 root row.

`zcat` / `gunzip` **are** in the headless allowlist — use them for the
`Packages.gz` / repodata pulls.  The discipline is *what* you pull, not
*whether you can*: pull only the kernel version, only when a verdict could
change.

**Never record NixOS channel git-revisions** (the
`channels.nixos.org/<channel>/git-revision` pins) in the tracker.  They
advance on nearly every run — recording them manufactures a diff on an
otherwise no-op run.

## Conventions for status entries

- `:white_check_mark:` — either the release's kernel carries the
  `736b380e28d0` backport (confirmed in changelog / advisory / kernel pin,
  not merely announced) — record the fixed kernel version and set *Fixed
  since* — **or** the kernel is outside the reachable window (< 6.6 or
  ≥ 7.2), i.e. *not affected*; note the reason ("trigger not present
  (< 6.6)" / "predates the bug" / "fixed upstream").
- `:x:` — vulnerable and **root-exploitable**: an in-window kernel without
  the backport on a distro whose `CONFIG_INIT_ON_ALLOC_DEFAULT_ON` is off
  (the EL10 family).
- `:warning:` — not fully resolved: either the fix is staged but not yet in
  the user-facing channel (merged upstream / cherry-picked but not in a
  released package); **or** the kernel is in-window and unpatched but a
  distro default reduces impact — `init_on_alloc=on` (DoS only) or
  unprivileged userns disabled.  A mitigation is **not** a fix — the kernel
  hole remains, so it never earns `:white_check_mark:`.
- `:grey_question:` — not yet verified (kernel pin, package version, or
  `init_on_alloc` default not yet inspected).
- `:heavy_minus_sign:` — used in the *Upstream fixed versions* table for a
  stable branch that predates the reachable window (6.1.x and older).

### "Fixed since" column

`Fixed since` is a **sticky first-observation** date: the date the
release's kernel fix first held.  Set it when a row flips to fixed; only
bump it if the verdict flips again or the fixed kernel version changes.
Leave it `—` while the row is vulnerable, DoS-only, or unverified — and for
"not affected" rows, which were never vulnerable.

### Verification log

When you re-verify entries, update the `## Verification log` section
rather than appending a line per re-check.  Bump the
`*Last verified <date>.*` line under the heading and edit the relevant
`### Upstream` / `### Distributions` subsection in place.  Add a new
`###` subsection only when a genuinely new topic appears.  This section
is also where auto-update-agent-relevant detail belongs — keep such
mechanics out of the tracker body.

The verified-date lives **only** in the `*Last verified <date>.*` header.
Do not repeat it inline — not in the tracker body, and not per bullet in
the log.  Every entry in the log is already "as of" that header date.
Method/source attribution *without* a date is fine and useful (e.g.
`(via …/madison)`, `(checked against ~/src/linux/stable)`).

### Date handling — first-seen / last-changed, not "today"

Dates embedded in the prose (front-matter `lastmod`, the verification log
header, and every "as of <date>" / "on <date>" / "released <date>" /
"Fixed since" value) are **first-seen / last-changed** dates, not
"today" dates.  Only move a date when the fact it qualifies actually
changes.  A re-verification that confirms the existing state is the
*purpose* of the `*Last verified <date>.*` header — that one line bumps;
nothing else does.  If the entire run is a no-op (no status, kernel
version, advisory, or upstream-backport change), leave the file alone and
don't commit at all — don't bump `lastmod`, don't bump `*Last verified*`,
and don't insert "re-confirmed <today>" parentheticals.  The next real
change is what records that everything in between was still current.

## Auto-update worktree

The systemd timer runs `scripts/auto-update` from a dedicated git
worktree at `~/src/auto-update/ipv6_frag_escape`, checked out on a single
long-lived branch named `auto-update`.  The wrapper merges `origin/main`
forward into that branch on each run, then hands off to headless Claude,
which commits any tracker changes back onto `auto-update` only.  The
agent must not create per-run branches, switch branches, push, or open
PRs — merges of `auto-update` into `main` are done manually by the user.

One-time setup (from the primary checkout at `~/src/ipv6_frag_escape`):

```
git worktree add -b auto-update ~/src/auto-update/ipv6_frag_escape main
```

The systemd units ship in `systemd/`.  They are not in a standard unit
search path, so wiring the timer means symlinking both units into
`~/.config/systemd/user/` and then enabling the timer.  Use `ln -sr` so
the links are relative — they survive the home directory being moved or
mounted at a different path:

```
ln -sr ~/src/ipv6_frag_escape/systemd/ipv6-frag-escape-tracker-update.service \
       ~/src/ipv6_frag_escape/systemd/ipv6-frag-escape-tracker-update.timer \
       ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable --now ipv6-frag-escape-tracker-update.timer
```

## Tearing down the auto-update

To stop the scheduled refresh entirely, unwire it in this order — the
sequence matters, because `systemctl disable` needs the unit definition
to still be resolvable when it runs.

1. **Disable the timer first**, while the unit symlinks are still in
   place:

   ```
   systemctl --user disable --now ipv6-frag-escape-tracker-update.timer
   ```

2. **Remove the unit-definition symlinks** from the search path:

   ```
   rm ~/.config/systemd/user/ipv6-frag-escape-tracker-update.timer \
      ~/.config/systemd/user/ipv6-frag-escape-tracker-update.service
   ```

3. **Reload** so the running user manager drops the units:

   ```
   systemctl --user daemon-reload
   ```

If the definition symlinks were removed *before* disabling, the stale
`enable` symlink is left behind, parking the timer in a `failed` state.
Recover by deleting it directly, then reloading and clearing the failed
residue:

```
rm ~/.config/systemd/user/timers.target.wants/ipv6-frag-escape-tracker-update.timer
systemctl --user daemon-reload
systemctl --user reset-failed ipv6-frag-escape-tracker-update.timer
```

Finally, remove the worktree and its branch (run from
`~/src/ipv6_frag_escape`):

```
git worktree remove ~/src/auto-update/ipv6_frag_escape
git branch -d auto-update
```

## Local kernel clone for git.kernel.org

git.kernel.org's cgit HTML pages (any URL ending in /log/, /tree/,
/commit/, etc.) are Anubis-gated; WebFetch hits the no-JS challenge and
the auto-update agent cannot read them.  The agent inspects kernel
history via long-living local clones under `~/src/linux/`:

| Clone path           | Upstream                                                           |
|----------------------|--------------------------------------------------------------------|
| `~/src/linux/stable` | `https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git` |
| `~/src/linux/net`    | `https://git.kernel.org/pub/scm/linux/kernel/git/netdev/net.git`   |
| `~/src/linux/vulns`  | `https://git.kernel.org/pub/scm/linux/security/vulns.git`          |

`net` is the netdev tree, where the fragmentation fixes land (and get
tagged for stable) before Linus.  `vulns` is the kernel CNA's CVE database
— empty for this bug until a CVE is assigned (see "The CVE, if one lands").

The auto-update *wrapper* refreshes the clones with `git -C <clone> fetch`
before invoking the headless agent; the agent inspects via the
`origin/<branch>` remote-tracking refs (e.g. `origin/linux-6.12.y`).

**Is a stable branch fixed yet?**  The backport is a cherry-pick with a new
SHA, so search by the upstream reference / subject, not `git tag
--contains`:

```
git -C ~/src/linux/stable log v<series>..origin/linux-<series>.y --grep=736b380e28d0 --grep='account for fraggap' --format='%h %s'
```

Empty output ⇒ the series is still unpatched.  Keep the range bounded to
`v<series>..` — unbounded, the subject grep matches a 2005-era commit
(*ip_output: account for fraggap when checking to add trailer_len*) and
reads as a false "fixed".  Confirm the fix landed
mainline in v7.2-rc1 with:

```
git -C ~/src/linux/stable describe --contains 736b380e28d0
```

The reachability boundary is the trigger `ce650a166335`: present in 6.6.y+
and absent from 6.1.y (a branch without it is not exploitable):

```
git -C ~/src/linux/stable log origin/linux-<series>.y --grep=ce650a166335 --format='%h %s'
```

The git smart-HTTP protocol is not Anubis-gated, so `git fetch` /
`git ls-remote` work from any UA.

## The CVE, if one lands

There is **no CVE** at seed time.  The kernel CNA's `vulns.git` keys every
record on the *fixing commit SHA*, and the fix commits are known, so the
check is a deterministic grep — no keyword guessing.  Inspect via
`origin/master`, not `HEAD` (the wrapper only `git fetch`es):

```
git -C ~/src/linux/vulns grep -l -e 736b380e28d0 -e eca856950f7c -e 38becddc origin/master -- 'cve/published/*'
```

A hit means a CVE was assigned; read its `.dyad` for the authoritative
per-branch `<introduced>:<fixed>` versions, add the CVE to the Summary and
References, and note it in the verification log.  The repo/site slug stays
`ipv6_frag_escape` regardless (see `WEBSITE.md`) — do not rename on CVE
assignment unless the user decides to.  A distro or MITRE CVE could also
appear in NVD without a `vulns.git` record — watch the disclosure sources
(AlmaLinux / TuxCare / oss-security) as a fuzzier secondary signal.

## init_on_alloc — the root-vs-DoS discriminator

For an in-window unpatched kernel, whether the row is `:x:` (root) or
`:warning:` (DoS) is set by the distro's `CONFIG_INIT_ON_ALLOC_DEFAULT_ON`
default (a boot-time `init_on_alloc=1` overrides a compiled-off default).
Known defaults: **off** on the EL10 family (root); **on** on Debian and
Ubuntu (DoS); **on** on NixOS — nixpkgs forces it in
`pkgs/os-specific/linux/kernel/common-config.nix`
(`INIT_ON_ALLOC_DEFAULT_ON = yes`):

```
git -C ~/src/nixos/nixpkgs grep -i INIT_ON_ALLOC origin/nixos-26.05 -- 'pkgs/os-specific/linux/kernel/*'
```

For a distro whose default is unknown, read the shipped kernel config
(`/boot/config-*`, the distro's `config-*` package, or the repodata
`kernel` build) for `CONFIG_INIT_ON_ALLOC_DEFAULT_ON`.  This posture is a
per-distro property that rarely changes; re-derive it only when adding a
distro or when a distro reworks its hardening defaults.

## Local nixpkgs clone for NixOS channel verification

*Seeding-and-adoption method — see "Routine run scope" above.  Do not
record the resolved channel revisions in the tracker.*

NixOS rows are verified from a local nixpkgs clone at `~/src/nixos/nixpkgs`,
not the (JS-rendered, lagging) security-tracker page.  Each channel has a
git-revision pointer at `https://channels.nixos.org/<channel>/git-revision`;
at that commit, `pkgs/os-specific/linux/kernel/kernels-org.json` carries
the version for every supported mainline.org kernel series.  The **default**
`linuxPackages` is the latest LTS (6.12.x); `linuxPackages_latest` is the
newest series (7.1.x at seed) — both are in-window and DoS-only, so the
verdict is the same either way.

Tracked channels: `nixos-26.05` (default), `nixos-unstable`.  (The
`-small` variants lead by a point release or two; add them only if a fix
ever lands and the lead matters.)

```
rev=$(curl -fsSL https://channels.nixos.org/<channel>/git-revision)
git -C ~/src/nixos/nixpkgs show "${rev}:pkgs/os-specific/linux/kernel/kernels-org.json"
```

`channels.nixos.org/<channel>/git-revision` returns a 302 — always pass
`-L` to curl.  The wrapper refreshes the clone on every run; for
interactive use, `git -C ~/src/nixos/nixpkgs fetch --quiet origin` first.

*Seeding-and-adoption method — see "Routine run scope" above.  The

pve-no-subscription → pve-enterprise, so no-subscription is strictly the
leading indicator and enterprise receives the same kernels later.

fetch that, decompress, and read the `<name>kernel</name>` package's
`<version ver= rel=>`.  The EL `os/` repos accumulate every point
release's kernel, so pick the numerically-highest `rel` (`sort -V`, not
string sort).

- **Rocky** BaseOS: `https://dl.rockylinux.org/pub/rocky/<8|9|10>/BaseOS/x86_64/os`.
  The dist tag encodes the minor (`el9_8` = 9.8, `el10_2` = 10.2).  Only
  **Rocky 10** (6.12, el10) is in-window and, being EL10 (`init_on_alloc`
  off), root-exploitable; Rocky 9 (5.14) and 8 (4.18) predate the bug.  The
  Rocky 10 row flips when its BaseOS kernel NVR reaches the RHEL 10 fixed
  build from the Red Hat record above (AlmaLinux, already advised, is the
  fastest cross-check).
- **Amazon Linux**: the ALAS signal is the repodata **`updateinfo.xml.gz`**
  (maps CVE → ALAS → fixed kernel NVR) — the per-CVE ALAS HTML pages are
  JS-rendered and return nothing headlessly.  AL2023's default `kernel` is
  6.1 (< 6.6, not affected); its opt-in `kernel6.12` / `kernel6.18` streams
  are in-window, so grep updateinfo for the CVE and check **each** stream;
  AL2 core `kernel` is 4.14 (not affected).  Resolve
  `…/al2023/core/mirrors/latest/x86_64/mirror.list` (AL2023) or
  `…/2/core/latest/x86_64/mirror.list` (AL2), fetch
  `<base>repodata/updateinfo.xml.gz` (grep the CVE) and `primary.xml.gz`
  (current versions).

## Debian kernel version source

**The security tracker, not a madison base-version compare, is authoritative for Debian status.** Two traps make a naive version check wrong: the base suite lags the `-security` upload that ships the fix, and Debian often backports the fix to a version *below* the upstream first-fixed release — so comparing a base-suite version against the upstream first-fixed release can read an already-fixed suite as vulnerable.

Read status and fixed version straight from the security tracker. WebFetch the human-readable per-CVE page, or `curl` the JSON and read the `CVE-2026-53362` block for each release's `status` (resolved/open) and the version under `repositories` (a `<suite>-security` entry means the fix shipped as a security update):

```
curl -fsSL 'https://security-tracker.debian.org/tracker/data/json'
```

Use the dak madison API only for the base-suite version and the sid/testing lineage (unstable=sid, testing=forky, stable=trixie, oldstable=bookworm, oldoldstable=bullseye):

```
curl -fsSL 'https://api.ftp-master.debian.org/madison?package=linux&s=sid,forky,trixie,bookworm,bullseye&text=on'
```

For a *Fixed since* date, use the `first_seen` of the fixed version in snapshot.debian.org:

```
curl -fsSL 'https://snapshot.debian.org/mr/package/linux/<version>/srcfiles?fileinfo=1'
```

Debian ships `init_on_alloc=on`, so in-window Debian kernels (trixie 6.12,
forky/sid 7.0) are DoS-only until the backport lands.

## Platform-specific notes

- **EL10 family (Rocky/RHEL/Alma/Oracle/CloudLinux 10):** the
  root-exploitable case — `CONFIG_INIT_ON_ALLOC_DEFAULT_ON` off,
  world-readable BTF, 6.12 kernel in-window.  AlmaLinux ships ahead of
  Rocky/Red Hat and is the leading indicator for the fixed kernel version.
  Rocky 8/9 (4.18/5.14) predate the bug.
- **Debian / Proxmox VE / Ubuntu:** `init_on_alloc=on` ⇒ in-window kernels
  are DoS-only (`:warning:`), not root.  Proxmox is Ubuntu-derived.
- **NixOS:** `init_on_alloc` forced on in nixpkgs ⇒ DoS-only.  Unprivileged
  userns on by default, so the residual DoS is reachable.  Determine the
  kernel pin per channel via the local nixpkgs clone (above).
- **Mitigation vs fix:** `init_on_alloc=1` (downgrades root→DoS) and
  disabling unprivileged userns (removes the trigger) both leave the kernel
  hole — record as `:warning:`, never `:white_check_mark:`.
